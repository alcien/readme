## 2. RAG 연구 동향 (2025–2026)

### 2.1 RAG의 위상 변화: "검색 보조"에서 "Context Engine"으로

2026년 현재, RAG는 단순한 검색-생성 파이프라인을 넘어서 엔터프라이즈 AI 아키텍처의 핵심 기반(Core Infrastructure)으로 자리 잡았다. 공공부문 AI(정책 QA, 행정 지식 관리)에서 환각 최소화 + 출처 제시의 핵심으로, RAGFlow의 2025년 연말 리뷰에 따르면 RAG는 "Retrieval-Augmented Generation"이라는 특정 패턴에서 "지능형 검색을 핵심 역량으로 하는 Context Engine"으로 진화 중이며 이 변화는 비가역적이다.

진화 경로: **Naive RAG → Advanced RAG → Modular RAG → Agentic RAG**

- 2024년부터 가열됐던 "긴 컨텍스트가 RAG를 대체할 수 있는가" 논쟁은 실무에서 "Lost in the Middle" 현상과 비용 문제로 RAG의 필요성이 재확인됨
- RAG가 2026 엔터프라이즈의 4대 우선순위(정확성, 설명가능성, 규정준수, 비용효율성)와 완벽 부합

### 2.2 Hybrid RAG (벡터 + GraphRAG) — 본 제안의 핵심 #1

RFP의 "벡터 인덱스 + Knowledge Graph"와 정확히 일치하는 최신 트렌드.

#### GraphRAG란?
전통적 RAG가 문서를 독립적 chunk로 처리하여 엔티티 간 관계를 놓치는 문제를 해결. Knowledge Graph의 노드-엣지 구조를 활용하여 multi-hop reasoning과 관계 기반 검색을 가능하게 한다.

#### 핵심 연구 성과
- **AAAI 2026**: "You Don't Need Pre-built Graphs for RAG" — 적응적 추론 구조를 동적 생성
- **ICLR 2026**: LinearRAG — relation-free 그래프 구축으로 효율적 GraphRAG
- **NeurIPS 2025**: GFM-RAG — 그래프 기반 검색의 foundation model 접근
- **CIKM 2025**: DocPolicyKG — **중국어 정책 문서**에서 경량 LLM 기반 KG 자동 구축 (의성군 행정 문서 적용 유사 사례)
- **GraphRAG Benchmark (ICLR 2026)**: 표준 평가 벤치마크 확립

#### Hybrid 전략의 성능
- NVIDIA/BlackRock 연구(2024~2025)와 2026 논문들에서 GraphRAG가 구조적 관계(정책·민원 절차) 추론에 강하고, 벡터가 의미 유사성을 보완하여 Faithfulness·Context Recall **10~20% 향상**
- GraphRAG 단독 vs Vector RAG 단독 vs Hybrid: 복잡 질의에서 Hybrid가 80% 정확도(Vector RAG 50%)
- LazyGraphRAG(Microsoft, 2025.06): 인덱싱 비용을 Full GraphRAG의 0.1%로 절감하면서 품질 유지

#### 의성군 적용 전략

| 서비스 | 검색 방식 | RFP 근거 |
|--------|----------|---------|
| 대민 - 정책·민원 안내 | **GraphRAG** (절차 그래프) + Vector | FUR-03: "절차 중심 답변 정확도" |
| 대민 - FAQ/일반 질의 | Vector RAG + BM25 | 단순 의미 검색, 저지연 |
| 직원 - 규정·매뉴얼 검색 | **Hybrid** (Graph + Vector + BM25) | 조례-규정-사례 관계 추론 필요 |
| 직원 - 문서 생성/요약 | Vector RAG | 문맥 기반 참조 문서 검색 |

권장 스택: **Vector DB(pgvector 또는 Milvus) + KG(Neo4j) 하이브리드 인덱싱**

### 2.3 Agentic RAG — 본 제안의 핵심 #2

단순 검색 → Agent가 동적으로 retrieval 계획/재시도/멀티홉 수행. RFP의 "Policy Router + Agent 기반 실행"에 직접 적용 가능.

#### A-RAG (arXiv 2602.03442, 2026.02)
- **Hierarchical Retrieval Interfaces**: 키워드 검색, 의미 검색, 청크 읽기를 에이전트에게 직접 노출
- Agent가 자율적으로 검색 전략을 선택하고, 반복적으로 증거를 수집하며, 충분할 때 최종 답변 생성
- HotpotQA 등 multi-hop QA 벤치마크에서 기존 방법 대비 일관적 우위, **94.5% 정확도** 달성 사례
- 전통적 RAG는 복잡 multi-hop 질의에서 ~34% 정확도 → Agentic RAG는 78%로 도약 (2026.03 SoK 논문)
- **Hierarchical Agentic RAG**: flat/단일 에이전트 대비 도메인 QA에서 **5~13 percentage points 개선**

#### LangGraph 기반 구현
LangGraph의 그래프 기반 state machine이 Agentic RAG의 구성요소 아키텍처에 자연스럽게 매핑:
- 각 구성요소(Router, Retriever, Grader, Generator, Hallucination Checker)가 노드
- 조건부 엣지가 의사결정 로직을 인코딩
- 상태 객체가 루프 전반에 걸쳐 컨텍스트를 누적

### 2.4 공공부문 RAG 적용 사례

- **E-Government GraphRAG (MDPI 2025.06)**: EU 집행위 보도자료 기반 multi-agent GraphRAG 프레임워크, embedding/graph/web search 파이프라인을 각각 독립 에이전트 도구로 설계. 오픈소스 공개.
- **SAS (2026 예측)**: "기관 지식 RAG로 신입 멘토링" — 행정 조직의 암묵지를 RAG로 구조화
- **OECD·프랑스 IGF**: 내부 정책 문서 RAG 시스템 운영
- **미국 정부 정책 QA (AGORA corpus)**: RAG가 정책 질의에서 신뢰도·정확도 크게 향상
- 공공 데이터(누리집, 백서, 조례)처럼 **구조화된 행정 문서에 GraphRAG가 최적**

### 2.5 RAG 평가 체계 (RFP QUR-01, QUR-02 대응)

| RFP 요구 지표 | 최신 권장 추가 |
|-------------|-------------|
| F1, Recall, Precision | Recall@k, MRR |
| BLEU, ROUGE | BERTScore (의미 유사도) |
| Faithfulness | RAGAS 프레임워크 (Context Precision, Context Recall, Answer Relevancy) |
| 골든 데이터셋 50건+ | 도메인별 세분화 + 리플레이 버퍼 겸용 |

**제안 시 강조**: Hybrid RAG + Agentic RAG 조합으로 RFP의 "절차 중심 답변 정확도 제고" + "근거 중심 응답 + 출처 제시"를 달성. 벤치마크 결과(Recall@k, Faithfulness) 제시 권장.

## 3. Fine-tuning & Catastrophic Forgetting 방지 — 통합 전략

### 3.1 RFP 요구사항 정리

> "LoRA, QLoRA 등 효율적 Fine-tuning 기법 적용"  
> "Fine-tuning 시 Catastrophic Forgetting 방지 방안 제시·적용"  
> "합성데이터 투입 비율 증가에 따른 단계별 성능 변화 추적"  
> "Model Collapse 징후 발생 시 이전 안정 버전으로 롤백"  
> "골든 데이터셋 50건 이상 구축" + "Fine-tuning 전후 성능 저하 추적"

### 3.2 핵심 연구 동향 (2025–2026)

#### CF 현상의 최신 이해
- IEEE TASLP 2025: 1B~7B LLM에서 CF 보편적, **모델 규모가 커질수록 forgetting 심각도 증가**
- ICLR 2025 "Spurious Forgetting": 성능 하락이 실제 지식 상실이 아닌 **태스크 정렬(Task Alignment) 저하**인 경우 다수 → 하위 레이어 동결(Freezing)로 개선
- Decoder-only 모델(BLOOMZ 등)이 Encoder-Decoder보다 forgetting 적음
- 행정 도메인(백서·조례·회의록)은 도메인 shift가 크므로 Continual Learning 필수

#### 주요 기법 분류

| 범주 | 기법 | 핵심 원리 | 의성군 적용성 |
|------|------|----------|-------------|
| **Replay 기반** | Hybrid-Confidence-Aware Selective Replay (2026 arXiv) | 이전 태스크 고신뢰 샘플만 재사용 → forgetting 최소화 + 컴퓨트 절감 | ◎ 핵심 |
| | Self-Synthesized Rehearsal | 합성 데이터로 과거 지식 보존 | ◎ (RFP 합성데이터 요구 부합) |
| **직교/정규화** | DOC Fine-tuning (ICLR 2025) | functional direction 동적 추적·보호 | ○ 권장 |
| | O-LoRA (2026) | LoRA + 직교성 제약, 이전 gradient 부분공간과 직교 학습 | ◎ 핵심 |
| **RL 기반** | SDFT (MIT 2026) | On-policy RL이 SFT보다 forgetting에 강인 | △ 참조 (2.5x 컴퓨트) |
| **Token 수준** | Low-Perplexity Token Learning (NeurIPS 2025) | LLM 생성 데이터로 fine-tuning 시 high-perplexity 토큰 감소 → CF 완화 | ○ 전처리에 적용 |
| **기타** | Momentum-Filtered Optimizer (MoFO) | 모멘텀 기반 중요 파라미터 보호 | △ 참조 |

### 3.3 제안 전략: Replay + Multi-LoRA + Drift 모니터링 3축 체계

**8개월 구축 기간 + 96GB VRAM 제약에서 확실하게 작동하는 실무 전략:**

#### 축 1: 골든 데이터셋 리플레이 (Rehearsal & Replay)

RFP가 명시한 "골든 데이터셋 50건+"을 **단순 평가용이 아닌, 학습 시 기존 지식을 보존하는 리플레이 버퍼로 적극 활용**.

- Fine-tuning 시 신규 도메인 데이터 + 골든셋 혼합 학습 (최적 비율 탐색: 80:20 → 70:30)
- 특히 수시 업데이트가 필요한 도메인에서 기존 지식과 신규 데이터를 혼합하여 지식 간섭과 Model Collapse 원천 차단
- 합성데이터(Synthetic Data) 기반 Replay: RFP의 비식별화 요구와 자연스럽게 결합 — 원본 개인정보 저장 없이 리플레이 가능
- 합성데이터 투입 비율 0%→10%→20%→30%→50% 단계적 증가, 각 단계 성능 추적 (RFP QUR-03 직접 대응)

#### 축 2: Multi-LoRA 기반 파라미터 모듈화 (Parameter Isolation)

45B 베이스 모델 가중치 **완전 동결** + 태스크별 독립 LoRA 어댑터:

```
[45B Base Model — 동결 (Frozen)]
    │
    ├── 요약 LoRA (r=16)
    ├── 번역 LoRA (r=16) 
    ├── 문서생성 LoRA (r=32)
    ├── 행정도메인 LoRA (r=32, QLoRA 4-bit)
    ├── 대민 서비스 LoRA (r=16)
    └── [신규 태스크] → 새 LoRA 추가 (기존 어댑터 무영향)
```

핵심 이점:
- **Zero-Forgetting**: 신규 태스크 LoRA가 추가되어도 기존 태스크 가중치는 전혀 오염되지 않음
- **vLLM 동적 어댑터 배칭**: GPU VRAM 단편화 없이 50명+ 동시 접속 처리 — LoRA 어댑터를 런타임에 동적 로드/언로드
- **RFP "Orchestrator 업무 유형 판별"과 직결**: LangGraph Orchestrator가 질의 유형 판별 → 적합한 LoRA 선택 → 라우팅
- Prompt Tuning의 장점(가벼운 전환)을 가져오면서 지식 주입 깊이 확보

#### 축 3: Drift 모니터링 — Fine-tuning 트리거 결정자

CF 방지의 세 번째 축은 "언제 Fine-tuning이 필요한가"를 정확히 판단하는 모니터링 체계. 불필요한 재학습을 방지하면서 품질 저하는 놓치지 않는 구조.

**4계층 Drift 모니터링:**

| 계층 | 모니터링 대상 | 도구 | 주기 |
|------|-------------|------|------|
| **L1 Infrastructure** | GPU 사용률, 메모리, 응답 지연, 토큰 사용량, 동시 접속 | Prometheus + Grafana | 실시간 |
| **L2 Retrieval Quality** | Recall@k, MRR, Embedding 안정성, NN Overlap, 코퍼스 갱신률 | Evidently + Golden Query | 주간 |
| **L3 Response Quality** | Faithfulness, Answer Relevancy, BERTScore, TTR, 환각 발생률 | RAGAS + DeepEval | 일간 (샘플) |
| **L4 Business Quality** | 만족도 트렌드, "근거 없음" 비율, 미답변 목록, 다빈도 키워드 변화 | RDB 통계 + 대시보드 | 일/주/월/분기 |

**Drift 대응 에스컬레이션:**

| 수준 | 조건 | 대응 | Fine-tuning? |
|------|------|------|------------|
| 🟢 Green | 정상 범위 | 대시보드 기록 | 불필요 |
| 🟡 Yellow | L3/L4 지표 10~15% 하락 | 프롬프트 조정, RAG 파라미터 튜닝 | 불필요 |
| 🟠 Orange | L2 지표 하락 (Recall@k < 0.85) | 지식베이스 재색인, 리랭커 가중치 조정 | 불필요 |
| 🔴 Red | L2+L3 동시 하락, 만족도 25%+ 하락 | **Fine-tuning 파이프라인 가동** (QLoRA + O-LoRA + Replay) → 평가 → 롤백/배포 | **이때만 가동** |

**Embedding Drift 방지 원칙** (RAG 품질 유지의 핵심):
1. 파이프라인 버전 고정 — 임베딩 모델·전처리·청킹을 하나의 버전으로 관리
2. 부분 재임베딩 금지 — 변경 시 전체 코퍼스 재임베딩 (부분 재임베딩은 drift의 #1 원인)
3. 벡터 프로비넌스 저장 — 모든 벡터에 (모델 버전, 전처리 해시, 타임스탬프) 메타데이터
4. 증분 업데이트 시 반드시 동일 파이프라인 버전으로 임베딩

**폐쇄망 호환 도구 스택:**
| 도구 | 역할 | 라이선스 | 폐쇄망 가능 |
|------|------|---------|-----------|
| Evidently AI | 데이터/모델 drift 탐지 | Apache 2.0 | ◎ 완전 로컬 |
| RAGAS | RAG 특화 평가 | Apache 2.0 | ◎ 완전 로컬 |
| DeepEval | 50+ 메트릭 | 오픈소스 | ◎ 로컬 가능 |
| LangFuse | 프롬프트 추적·평가 | MIT | ◎ 셀프호스팅 |

### 3.4 통합 스토리라인

> "과적합과 망각은 **리플레이**로 막고, 기능 확장은 **Multi-LoRA**로 격리하며, Fine-tuning 시점은 **Drift 모니터링**이 결정하고, 전체 흐름은 **LangGraph**가 지휘한다."

### 4.2 Agentic RAG Pipeline 상세

```
[질의 입력]
     │
     ▼
[Query 전처리]
  ├── Query Expansion (동의어, 행정 용어 확장)
  ├── Query Classification (단순 사실 vs multi-hop vs 절차형)
  └── 하위 질의 분해 (A-RAG의 Hierarchical Retrieval 패턴)
     │
     ▼
[Agentic Retrieval Loop] ← Agent가 동적으로 전략 선택
  ├── Keyword Search (BM25) — 행정 용어 정확 매칭
  ├── Semantic Search (Vector DB) — 의미 유사도
  ├── Graph Traversal (Neo4j Cypher) — 관계/절차 추론
  └── Web Search (대민용, RAG 미보유 자료)
     │
     ├── Relevance Grading ← 검색 결과 품질 평가
     │   └── 불충분 → 재검색 (다른 전략으로)
     │
     ▼
[Reciprocal Rank Fusion] — 다중 검색 결과 통합
     │
     ▼
[Reranker] — Cross-encoder 기반 재순위화
     │
     ▼
[Context Window 구성]
  ├── 문서 chunk + 메타데이터 (출처, 작성일, 부서)
  ├── 그래프 컨텍스트 (엔티티 관계, 절차 경로)
  └── 이전 대화 맥락 (멀티턴)
     │
     ▼
[LLM 생성] → [Hallucination Check] → [출처 검증]
  └── 미통과 → 재생성 또는 "근거 자료 없음" 안내
```

### 4.3 Agent Manager 설계

| 구분 | Agent | 연계 시스템 | 기능 |
|------|-------|-----------|------|
| 대민 | 요금조회 Agent | 상하수도 요금 시스템 | 조회, 납기일, 체납, 감면 안내 |
| 대민 | 시설예약 Agent | 통합예약시스템 | 조회, 예약 신청 |
| 직원 | 회의실예약 Agent | 내부 예약시스템 | 조회, 예약, 취소, 알림 |
| 직원 | 인수인계 Agent | RAG + 문서생성 | 자동 문서 생성, 관련 자료 검색, AI 선임자 Q&A |
| 공통 | 데이터조회 Agent | 데이터정보센터 | 내부 데이터 연계 조회 |

```
[트리거 조건]

     │
     ▼
```

## 6. 결론 — 제안 전략 요약

### 6.1 핵심 아키텍처 한 줄 요약

> **"Hybrid RAG(Vector+Graph+BM25)로 검색하고, Multi-LoRA로 격리된 태스크별 어댑터를 LangGraph Orchestrator가 동적 라우팅하며, Drift 모니터링이 Fine-tuning 시점을 결정하고, 골든 데이터셋 Replay로 기존 지식을 보존한다."**

### 6.2 RFP 평가 항목별 대응 매핑

| RFP 요구 | 대응 전략 | 배점 영향 |
|---------|----------|---------|
| FUR-01: CF 방지 | Multi-LoRA 격리 + 골든셋 Replay + O-LoRA | 기능 8점 |
| FUR-02: LLMOps | 4계층 Drift 모니터링 + 자동 평가/롤백 | 기능 8점 |
| FUR-03: 대민 RAG | Hybrid RAG (GraphRAG + Vector + BM25) | 기능 8점 |
| FUR-05: RAG 체계 | Agentic RAG (A-RAG 패턴) + LangGraph | 기능 8점 |
| FUR-06: Agent | LangGraph Orchestrator + 5개+ 시스템 연계 | 기능 8점 |
| QUR-01~03: 품질 | RAGAS + 골든셋 50건 + TTR + 자동 롤백 | 품질 5점 |
| AI 모델 실증 | 45B+ 한국어 벤치마크 + 실시간 시연 | 실증 15점 |

### 6.3 차별화 포인트

1. **"Zero-Forgetting by Design"**: CF를 사후 방지가 아닌, Multi-LoRA 모듈 격리로 구조적 원천 차단
2. **골든 데이터셋의 이중 활용**: 평가용 + 리플레이 버퍼 겸용 — RFP 요구사항을 초과 충족
3. **Drift 기반 Fine-tuning 트리거**: 불필요한 재학습 방지 + 품질 저하 조기 포착
4. **Agentic RAG**: 전통 RAG 34% → 78% 정확도 도약 (2026 SoK 논문), A-RAG 94.5% HotpotQA
5. **E-Government GraphRAG**: 공공부문 전용 사례(MDPI 2025) 직접 참조
6. **LangGraph 통합**: Orchestrator + LoRA Router + Agent Manager를 단일 프레임워크로 지휘
